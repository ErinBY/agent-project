# ModoAI 수정 에이전트 설계 PRD (원고 v2 · 완성본)

> 작성: 변세영 · 유니드컴즈 서비스 기획자(AI Native Product) 포트폴리오용
> 상태: 섹션 1~7 확정 (v2에서 재시도 상한 2회로 표기 통일)

**개선 과제 한 문장**
모호한 수정 요청을 실행 전에 구체화하고, 실행 후에는 해석이 의도와 맞았는지 제품이 스스로 검증하는 '수정 에이전트'를 설계한다.

---

## 섹션 1. 개선 과제 정의

ModoAI의 이미지 수정은 텍스트 입력창 하나로 여러 요청을 한 번에 받습니다. 직접 써보면서 그릇 교체, 테이블보 변경, 소품 추가까지 6가지를 한 번에 요청해봤는데, 솔직히 반영 품질은 기대 이상이었습니다. 문제는 반영이 아니라 다른 곳에 있었습니다.

요청을 입력하다 보니 "테이블보를 좀 더 밝은 것으로", "그릇은 좀 더 모던한 것으로" 같은 표현을 쓰게 됩니다. 저도 그랬고, 대부분의 셀러가 그럴 겁니다. 디자이너가 아닌 이상 머릿속 이미지를 정확한 말로 옮기기 어렵기 때문입니다. 그런데 지금 구조에서는 AI가 '밝은'과 '모던한'을 어떻게 해석했는지 실행 전에는 알 수 없고, 결과는 크레딧이 차감된 뒤에야 확인할 수 있습니다. 제 의도와 AI의 해석 사이의 간극을 미리 좁힐 장치가 없는 겁니다.

구조를 뜯어보면 원인은 세 가지입니다. 요청을 항목 단위로 나누지 않아서 어떤 항목이 구체적이고 어떤 항목이 모호한지 시스템이 구분하지 못합니다. 모호한 항목을 실행 전에 구체화하는 단계(선택지를 보여주거나 되묻는 과정)가 없어서, 해석이 전부 모델 안에서만 일어납니다. 그리고 크레딧이 결과가 아니라 시도에 차감되기 때문에, "생각과 다르네 → 다시 요청 → 또 차감"이라는 반복의 비용을 사용자가 부담합니다.

이 문제가 민감한 이유는 발생하는 위치가 결제 접점이기 때문입니다. 크레딧은 ModoAI의 수익 모델인데, "몇 번을 써야 원하는 게 나올지 모르겠다"는 불확실성이 바로 그 위에 얹혀 있습니다. 표현을 잘하는 사용자와 못하는 사용자의 결과 차이도 벌어지는데, ModoAI의 타깃인 1인 셀러는 대부분 후자입니다. 'No Prompt, No Stress'를 내건 제품이 수정 단계에서는 프롬프트 실력을 요구하게 되는 셈입니다. 원하는 결과까지 시도가 늘어날수록, ModoAI는 비즈니스 파트너가 아니라 뽑기 도구로 경험됩니다.

9년간 CS 문의 패턴에서 제품이 풀어야 할 문제를 찾아왔습니다. 이번에는 사용자로서 직접 겪은 마찰을 같은 방식으로 과제로 정의했습니다.

**개선 과제: 모호한 수정 요청을 실행 전에 구체화하고, 실행 후에는 해석이 의도와 맞았는지 제품이 스스로 검증하는 '수정 에이전트'를 설계한다.**

*(첨부 예정 캡처: ① 6개 항목을 입력한 수정 요청 화면 ② 수정 결과 이미지 ③ 크레딧 차감 안내)*

## 섹션 2. 에이전트의 목표와 성공 기준

**목표: 셀러가 프롬프트 실력 없이도, 최소한의 크레딧으로 의도한 결과에 도달하게 한다.**
모호함은 실행 전에 풀고, 해석이 맞았는지는 사용자가 아니라 제품이 확인합니다. 실패의 비용을 사용자에게 넘기지 않는 것이 이 에이전트의 원칙입니다.

| 지표 | 정의 | 방향 |
|---|---|---|
| 첫 시도 의도 일치율 | 첫 수정 결과를 사용자가 그대로 수락(다운로드·다음 단계 진행)한 비율 | 올라야 함 |
| 목표 도달 크레딧 수 | 최종 만족한 결과까지 쓴 평균 크레딧 | 내려야 함 |
| 동일 대상 재수정률 | 같은 요소(예: 테이블보)를 연속으로 다시 고친 비율 — 해석이 빗나갔다는 신호 | 내려야 함 |
| 구체화 단계 응답률 | AI가 보여준 선택지에서 사용자가 실제로 고른 비율 — 되묻기가 도움인지 마찰인지 확인 | 올라야 함 |
| 수정 기능 재사용률 | 수정을 써본 사용자가 다음 촬영에서도 수정을 쓰는 비율 | 올라야 함 |

구체화 단계 응답률을 지표에 넣은 이유가 있습니다. 되묻기가 너무 잦으면 그 자체가 마찰이 되기 때문입니다. 이 숫자가 낮게 나오면 되묻는 기준을 완화하라는 신호로 씁니다. 편의를 위해 만든 장치가 불편이 되고 있지 않은지, 장치 스스로도 검증받게 했습니다.

## 섹션 3. Agentic Workflow 설계 — 수정 에이전트는 어떤 순서로 일하는가

사용자가 수정 요청을 입력한 순간부터 결과를 받기까지, 에이전트는 6단계로 일합니다. 핵심은 두 개의 관문입니다. 실행 전에는 모호함을 걸러내고(1~3단계), 실행 후에는 해석이 맞았는지 스스로 확인합니다(5~6단계). 지금의 ModoAI에는 이 두 관문이 없어서, 그 역할을 사용자가 크레딧을 써가며 대신하고 있습니다.

### 플로우차트 구조

```
사용자 수정 요청 입력
        │
[1] 요청 분해 ─── 자유 텍스트를 항목 단위로 쪼갬
        │
[2] 모호성 판정 ─ 항목마다 "이대로 실행 가능한가?" 판단
        │
        ├─ 명확한 항목 ──────────────┐
        │                            │
        └─ 모호한 항목               │
                │                    │
[3] 구체화 ── 선택지 칩 제시         │
     "밝은 테이블보란?"              │
     → 화이트 린넨 / 크림 / 그레이   │
     사용자가 탭해서 선택            │
                │                    │
                └────────┬───────────┘
                         │
[4] 실행 ── 확정된 항목들로 이미지 편집 모델 호출
                         │
[5] 검증 ── 결과 이미지를 항목별로 다시 확인
     · 요청이 반영됐는가?
     · 제품 원형은 보존됐는가?
                         │
        ├─ 전부 통과 → [6a] 전달 + 크레딧 차감 → 종료
        │
        └─ 일부 미달 → 실패 항목만 자동 재시도 (최대 2회)
                │
                ├─ 재시도 성공 → [6a] 전달 + 크레딧 차감
                │
                └─ 재시도 실패 → [6b] 사용자에게 상황 보고 + 선택지 제시
                                  (이대로 받기 / 다시 시도 / 취소)
```

### 단계별 상세 표

| 단계 | AI가 보는 것 (입력) | AI가 판단하는 것 | AI가 하는 것 (실행) | 실패하면 |
|---|---|---|---|---|
| 1. 요청 분해 | 사용자가 입력한 자유 텍스트 | 요청이 몇 개의 독립된 항목인지 | "테이블보 밝게 + 그릇 교체 + 나이프 추가" → 항목 3개로 분리해 사용자 화면에 칩으로 표시 | 분해가 어려운 문장은 통째로 모호 항목 처리 → 2단계로 |
| 2. 모호성 판정 | 분해된 항목 각각 | 이 표현만으로 해석이 하나로 정해지는가 ("나이프 추가"=명확, "밝게"=모호) | 명확 항목은 실행 대기, 모호 항목은 3단계로 | 판정이 애매하면 모호 쪽으로 분류 (되묻는 비용 < 재생성 비용) |
| 3. 구체화 | 모호 항목 + 현재 이미지 | 이 항목의 해석 후보는 무엇인가 | 시각적 선택지 칩 2~4개 제시 (예: "밝은 테이블보" → 화이트 린넨/크림/라이트 그레이), 사용자가 선택 | 사용자가 선택을 건너뛰면 AI 추천안을 기본값으로 실행하되, 어떤 해석을 택했는지 표시 |
| 4. 실행 | 확정된 항목 목록 + 원본 이미지 | 항목 간 충돌·순서 (배경 교체 후 소품 추가 등) | 이미지 편집 모델 호출 | 모델 호출 자체가 실패하면 크레딧 차감 없이 재호출 |
| 5. 검증 | 수정 전후 이미지 + 확정 항목 목록 | 두 가지 축으로 항목별 판정 — ① 요청이 반영됐는가 ② 제품 원형·품질은 보존됐는가 (형태 왜곡, 결함 등) | 항목별 통과/미달 기록 | 미달 항목만 4단계로 자동 재시도 (최대 2회, 사용자 크레딧 차감 없음) |
| 6. 전달 | 검증 결과 | 전부 통과 vs 일부 미달 | (a) 통과: 결과 전달 + 이때 크레딧 차감 (b) 미달: "3개 중 2개 반영, 1개는 실패" 상황 보고 + 이대로 받기/다시 시도/취소 선택지 | — |

설계에서 가장 고민한 지점은 3단계 구체화입니다. 모든 항목을 되물으면 그 자체가 마찰이 되기 때문에, 되묻는 기준을 "해석이 하나로 정해지지 않는 항목"으로 한정했습니다. "나이프를 추가해줘"는 묻지 않고 바로 실행합니다. "밝게 해줘"만 묻습니다. 그리고 이 기준이 실제로 적절한지는 섹션 2의 '구체화 단계 응답률' 지표로 계속 검증합니다.

크레딧 차감 시점도 의도적으로 뒤로 옮겼습니다. 지금은 시도하면 차감이지만, 이 설계에서는 검증을 통과한 결과가 전달될 때 차감됩니다. 검증과 재시도에 드는 비용을 사용자가 아니라 제품이 부담하는 구조인데, 그만큼 재시도 횟수에 상한(2회)이 반드시 필요합니다. 이 원가 문제는 섹션 7에서 다룹니다.

## 섹션 4. 구성요소 정의 — 에이전트에게 무엇을 쥐여줄 것인가

섹션 3이 에이전트가 일하는 '순서'였다면, 이 섹션은 일을 시키기 위해 쥐여줘야 할 '재료'입니다. 신입 직원에게 일을 맡길 때 필요한 것들과 같습니다. 업무에 필요한 자료(Context), 단골 취향 노트(Memory), 쓸 수 있는 프로그램(Tool), 검수 기준(Evaluation), 그리고 "이건 꼭 물어보고 해"라는 선(Human-in-the-loop)입니다.

### Context — 일 시작 전에 책상에 올려줘야 할 자료

수정 작업 하나를 맡길 때마다 에이전트에게 네 가지를 줍니다. ① 원본 제품 사진 ② 지금 고치려는 이미지 ③ 사용자가 입력한 요청 ④ 처음 촬영 때 고른 컨셉 정보.

④를 주는 이유는 같은 요청도 컨셉에 따라 정답이 달라지기 때문입니다. "밝은 테이블보로 바꿔줘"라는 요청에, 소품과 온기가 있는 '브런치' 컨셉이라면 아이보리 린넨처럼 따뜻한 톤의 밝음이 어울리고, 장식을 덜어낸 '미니멀' 컨셉이라면 무늬 없는 순백처럼 깔끔한 톤의 밝음이 어울립니다. 같은 "밝게"라도 보여줄 선택지가 달라지는 것이고, 컨셉을 모르면 엉뚱한 선택지를 내밀게 됩니다.

그리고 자료에 경고장 하나를 함께 붙입니다. **"제품은 건드리지 마."** 배경, 소품, 테이블보는 바꿔도 되지만, 셀러가 파는 에그타르트 자체가 다른 모양이 되면 그 이미지는 쓸 수 없기 때문입니다.

### Memory — 단골 취향 노트: 뭘 적고, 뭘 안 적을 것인가

카페 직원이 단골의 "아이스, 샷 추가"를 기억하듯, 에이전트도 사용자의 선택을 기억합니다. 재료는 구체화 단계에서 고른 단어 칩입니다. "밝은"에서 아이보리를 고르고 "모던한"에서 무광 그릇을 고른 기록이 쌓이면, 다음부터는 그 취향을 첫 번째 선택지로 올려줍니다.

문제는 뭘 적을지입니다. 다 적으면 안 됩니다. 어제 딱 한 번 "크리스마스 느낌으로"를 골랐다고 그걸 취향으로 적어버리면, 1월에도 크리스마스 선택지를 내미는 직원이 됩니다. 그래서 규칙을 둡니다. **같은 계열의 선택이 3번 쌓이면 노트에 적고(취향으로 인정), 한 번뿐인 선택은 그날만 쓰고 버립니다.**

노트에 적힌 뒤에도 장치가 하나 더 필요합니다. 선택지에 항상 다른 분위기의 옵션을 하나 섞어두는 겁니다. 취향은 편하라고 있는 기본값이지, 사용자를 가두는 틀이 아니기 때문입니다. JD의 "기억하되, 갇히지 않게"라는 질문에 대한 제 답이 이 두 규칙입니다. **적는 조건(3회)과 빠져나갈 문(다른 결 옵션 상시 노출)을 같이 설계하는 것.**

### Tool Calling — 직원이 쓰는 프로그램 목록

에이전트는 스스로 그림을 그리지 못합니다. 판단을 하고, 프로그램을 부릅니다. 이 에이전트가 부르는 프로그램은 세 개입니다. ① 이미지를 실제로 고치는 편집 모델 ② 고쳐진 이미지를 검사하는 비전 모델(그리는 도구와 검사하는 도구를 왜 나누는지는 섹션 7에서 설명합니다 — 비용 때문입니다) ③ 크레딧을 차감하는 결제 API(검사 통과 후에만 호출).

도구도 실패할 수 있습니다. 모델 서버가 응답하지 않거나 오류를 반환하는 경우인데, 이는 결과 품질의 문제가 아니라 호출 자체의 실패이므로 크레딧 차감 없이 처리해야 합니다. 도구별 호출 실패 시의 동작은 섹션 5의 정책 표에 정의합니다.

### Evaluation — 검수 기준: 뭐가 '잘한 일'인가

검수 기준이 없으면 신입 직원은 자기가 잘했는지 모릅니다. 이 에이전트의 검수는 딱 두 가지를 봅니다.

**① 시킨 건 됐나.** "나이프 추가해줘"라고 했으면 결과에 나이프가 있는지 확인합니다. 항목별로 하나씩 봅니다.
**② 안 시킨 걸 건드렸나.** 제품의 모양·색이 변하지 않았는지, 손가락 6개 같은 AI 특유의 결함이 생기지 않았는지 봅니다.

둘 다 봐야 하는 이유는 간단합니다. ①만 보면 "나이프는 넣었는데 에그타르트가 찌그러진" 이미지가 통과되고, ②만 보면 "제품은 멀쩡한데 나이프는 무시한" 이미지가 통과됩니다. **두 검사를 모두 통과한 것만 '성공'** — 이게 JD의 "성공을 어떻게 정의할까요?"에 대한 제 답입니다.

### Human-in-the-loop — "이건 꼭 물어보고 해"라는 선

아무리 일 잘하는 직원도 마음대로 하면 안 되는 일이 있습니다. 이 에이전트가 반드시 사용자에게 물어보는 지점은 세 곳입니다.

① **실행 전** — 요청이 모호하면 해석 선택지를 보여주고 고르게 합니다. (구체화 칩)
② **실행 후** — 재시도까지 했는데도 실패한 항목은 어떻게 할지 사용자가 정합니다. (이대로 받기 / 다시 시도 / 취소)
③ **위험한 요청** — "제품 색을 바꿔줘"처럼 제품 자체를 바꾸는 요청은 바로 실행하지 않고, "실제 상품과 달라질 수 있어요"라고 확인받습니다.

세 지점의 공통 원칙은 하나입니다. **돈(크레딧)이 걸린 결정과 제품이 걸린 결정은 AI가 대신하지 않는다.** 자동화는 사용자의 수고를 줄이는 것이지, 결정권을 가져가는 게 아니기 때문입니다.

## 섹션 5. 정책·예외 케이스 — 틀리면 어떻게 되는가

에이전트 설계에서 행복한 경로(모든 게 잘 되는 흐름)는 절반일 뿐입니다. 나머지 절반은 틀렸을 때의 규칙입니다. 특히 이 에이전트는 크레딧이라는 돈이 걸려 있어서, 모든 예외 케이스에 크레딧 처리 방침을 함께 정의했습니다. 원칙은 섹션 2에서 세운 그대로입니다. **비용은 성공한 결과에만 발생한다.**

### A. 품질 실패 — AI가 일을 못 했을 때

| 케이스 | 시스템 동작 | 사용자에게 보이는 것 | 크레딧 |
|---|---|---|---|
| 일부 항목만 검증 통과 | 실패 항목만 자동 재시도 (최대 2회) | 항목별 상태값 표시: "그릇 교체 ✓ / 테이블보 변경 ✓ / 나이프 추가 진행 중" | 이 단계에서는 차감 없음 |
| 재시도에도 일부 실패 | 성공분만 반영된 결과 + 선택지 제시 | "3개 중 2개가 반영됐어요. ① 이대로 받기 ② 실패 항목만 다시 시도 ③ 취소" | ① 부분 차감(반영 비율 기준) ② 추가 차감 없이 1회 기회 ③ 차감 없음 |
| 전 항목 실패 | 결과를 전달하지 않고 종료 | "이번 수정은 반영에 실패했어요. 크레딧은 차감되지 않았습니다" + 요청 표현을 바꿔볼 것을 제안 | 차감 없음 |
| 검증은 통과했는데 사용자가 보기엔 아님 (검증 AI의 오판) | '결과가 달라요' 버튼으로 이의 접수 → 해당 건 크레딧 반환 + 오판 사례를 검증 기준 개선 데이터로 적재 | 결과 화면에 상시 노출되는 '결과가 달라요' 버튼 | 반환 |

### B. 요청 자체의 문제 — 입력이 애매하거나 위험할 때

| 케이스 | 시스템 동작 | 사용자에게 보이는 것 | 크레딧 |
|---|---|---|---|
| 모호한 항목 (해석이 여러 개) | 실행 보류, 구체화 칩 제시 | "'밝게'를 이 중에서 골라주세요: 아이보리 린넨 / 순백 / 라이트 그레이 / 직접 입력" | 실행 전이므로 차감 없음 |
| 구체화 칩에서 선택 안 하고 진행 | AI 추천안(1순위 칩)으로 실행하되, 어떤 해석을 택했는지 결과에 표시 | "테이블보는 '아이보리 린넨'으로 해석해 반영했어요" | 정상 차감 (검증 통과 시) |
| 항목이 서로 충돌 ("배경을 어둡게 + 전체적으로 밝게") | 실행 보류, 충돌 사실을 알리고 우선순위 질문 | "두 요청이 서로 반대돼요. 어느 쪽을 우선할까요?" | 실행 전이므로 차감 없음 |
| 제품 자체를 바꾸는 요청 ("에그타르트를 딸기타르트로") | 자동 실행하지 않고 경고 후 의사 확인 | "실제 판매 상품과 달라질 수 있어요. 그래도 진행할까요?" | 동의 시에만 정상 차감 |
| 수정 불가능한 요청 (지원 범위 밖) | 실행하지 않고 사유 안내 | "이 요청은 아직 지원하지 않아요" + 지원 가능한 수정 유형 안내 | 차감 없음 |

### C. 시스템 장애·중단 — 도구가 실패하거나 흐름이 끊겼을 때

| 케이스 | 시스템 동작 | 사용자에게 보이는 것 | 크레딧 |
|---|---|---|---|
| 편집 모델 호출 실패 (무응답·오류) | 자동 재호출 1회 → 그래도 실패 시 작업 중단 | "일시적인 오류예요. 잠시 후 다시 시도해주세요" | 차감 없음 |
| 검증 모델 호출 실패 | 검증 없이 전달하지 않음. 재호출 후에도 실패 시, 결과를 '검증 대기' 상태로 보관하고 사용자에게 안내 | "결과 확인이 지연되고 있어요. 완료되면 알려드릴게요" | 검증 완료 후 차감 |
| 크레딧 차감 실패 (이미지는 전달됨) | 차감 재시도를 백그라운드로 처리, 사용자 흐름은 막지 않음 | 사용자에게는 정상 완료로 보임 | 지연 차감 (내부 재처리) |
| 차감됐는데 이미지 전달 실패 | 전달 재시도 → 실패 시 자동 크레딧 복구 | "오류가 발생해 크레딧을 돌려드렸어요" | 자동 반환 |
| 크레딧 잔액 부족 | 실행 전에 차단 (실행 후 발견되는 일이 없어야 함) | 요청 입력 시점에 "크레딧이 부족해요" + 충전 안내 | — |
| 진행 중(실행·검증·재시도) 사용자가 창을 닫거나 이탈 | 작업은 서버 기준으로 계속 진행해 완주. 완료된 결과는 촬영 목록에 보관 | 재접속 시 "진행 중이던 수정이 완료됐어요" 알림, 촬영 목록에서 결과 확인 | 검증 통과분만 정상 차감. 이탈로 인한 결과 유실·이중 차감 없음 |

표를 관통하는 규칙은 네 줄로 요약됩니다.

1. **실행 전에 막을 수 있는 문제는 실행 전에 막는다.** 모호·충돌·잔액 부족은 크레딧을 쓰기 전에 해소합니다.
2. **제품의 실패 비용을 사용자에게 넘기지 않는다.** 품질 실패와 장애는 차감 없음 또는 반환이 기본입니다.
3. **검증 AI도 틀릴 수 있음을 인정한다.** '결과가 달라요' 경로를 상시로 열어두고, 오판 사례는 검증 기준을 고치는 재료로 씁니다.
4. **작업의 기준은 화면이 아니라 서버다.** 사용자가 창을 닫아도 진행 중인 작업은 완주시키고 결과를 보관합니다. 크레딧을 쓴 작업이 이탈로 증발하는 일이 없어야 하기 때문입니다.

## 섹션 6. 화면 설계 — 사용자 눈에는 어떻게 보이는가

에이전트가 아무리 잘 일해도, 사용자가 보는 건 화면뿐입니다. 이 설계의 핵심은 에이전트의 '상태'가 화면에 드러나는 방식입니다. 지금의 ModoAI 수정은 입력창 하나와 결과 이미지뿐이라 중간이 보이지 않습니다. 이 에이전트는 중간을 보여줍니다. 핵심 화면은 세 개입니다.

### 화면 1 — 수정 요청 입력: 내 요청이 어떻게 이해됐는지 보인다

사용자가 자유 텍스트를 입력하면, 전송 즉시 요청이 항목 칩으로 분해되어 입력창 아래 표시됩니다. "그릇은 모던하게, 테이블보는 밝게, 나이프 추가해줘"를 입력하면 [그릇 교체] [테이블보 변경] [나이프 추가] 세 개의 칩이 생기는 식입니다. 모호한 항목의 칩에는 물음표 배지가 붙고, 탭하면 구체화 선택지(아이보리 린넨 / 순백 / 라이트 그레이 / 직접 입력)가 펼쳐집니다. 명확한 항목은 배지 없이 바로 실행 대기 상태가 됩니다.

- 상태값(칩 단위): `명확(즉시 실행 가능)` / `모호(선택 대기)` / `충돌(우선순위 질문)` / `제품 변경(경고 확인 대기)` 4종
- 설계 의도: 실행 버튼은 모든 칩이 확정 상태가 되어야 활성화됩니다. 크레딧이 쓰이기 전에 해석을 확정 짓는 구조를 화면으로 강제하는 것입니다.

### 화면 2 — 진행 상태: 에이전트가 지금 뭘 하고 있는지 보인다

실행이 시작되면 각 칩이 진행 카드로 바뀌고, 항목별로 상태가 실시간 갱신됩니다. "그릇 교체 ✓ 완료 / 테이블보 변경 ✓ 완료 / 나이프 추가 ⟳ 다시 시도 중 (1/2)"처럼요. 검증 단계도 숨기지 않습니다. "결과를 확인하고 있어요"라는 문구로 에이전트가 스스로 검사 중임을 보여줍니다.

- 상태값(항목 단위): `실행 중` / `검증 중` / `완료` / `재시도 중(n/2)` / `실패` 5종
- 상태값(전체 작업 단위): `진행 중` / `완료` / `부분 완료` / `중단` 4종
- 설계 의도: 재시도가 몇 번째인지(1/2)까지 보여주는 이유는, 기다림에 근거를 주기 위해서입니다. 이유 없는 로딩은 불안이지만 "다시 시도 중 1/2"는 과정입니다. 사용자가 창을 닫아도 이 상태는 서버에 남아, 재접속 시 같은 화면으로 복원됩니다(섹션 5의 이탈 정책).

### 화면 3 — 결과·부분 실패: 무엇이 됐고 무엇이 안 됐는지 보인다

전부 성공하면 결과 이미지와 함께 항목별 반영 내역이 요약됩니다("3개 요청이 모두 반영됐어요 · 크레딧 1 차감"). AI 추천안으로 자동 진행된 항목이 있다면 "테이블보는 '아이보리 린넨'으로 해석했어요"를 함께 표시합니다.

부분 실패 시에는 성공분이 반영된 이미지 위에 실패 항목을 명시하고, 세 개의 선택 버튼을 제시합니다. [이대로 받기 — 크레딧 일부 차감] [실패 항목만 다시 시도 — 추가 차감 없음] [취소 — 차감 없음]. 버튼마다 크레딧 처리가 함께 적혀 있어, 선택의 비용을 읽고 결정할 수 있습니다. 결과 화면 하단에는 '결과가 달라요' 버튼이 상시 노출됩니다(검증 오판 이의 경로).

- 상태값: `전체 성공` / `부분 실패(선택 대기)` / `사용자 확정(수락·재시도·취소)` / `이의 접수` 4종
- 설계 의도: 실패를 숨기지 않고 선택지로 바꾸는 것. "안 됐다"로 끝나면 불만이지만, "여기까지 됐고, 다음은 당신이 정한다"면 통제감이 됩니다.

*(Figma Make 제작 시: 화면 3개를 각 1프레임씩, 화면 2는 재시도 중 상태로, 화면 3은 부분 실패 상태로 그리는 것이 설계 의도가 가장 잘 드러납니다.)*

## 섹션 7. 원가·운영 관점 — 이걸 돌리면 돈이 얼마나 드는가

AI는 부를 때마다 회사가 비용을 냅니다. 이 에이전트는 요청 분해, 모호성 판정, 실행, 검증, 재시도까지 AI를 여러 번 부르는 설계라서, 사용자를 편하게 해주는 만큼 회사의 호출 비용이 늘어납니다. 그래서 어디에 돈을 쓰고 어디서 아낄지를 함께 정하지 않으면 이 기획은 완성이 아닙니다. 판단은 네 가지입니다.

### ① 검사는 가벼운 모델에게, 생성은 무거운 모델에게

식당에서 요리는 셰프가 하지만, 주문대로 나왔는지 확인하는 일은 홀 직원도 할 수 있습니다. 확인까지 셰프에게 시키면 인건비 낭비입니다. AI도 같습니다. 이미지를 만드는 데는 크고 비싼 생성 모델이 필요하지만, "나이프가 들어갔나, 제품이 찌그러지지 않았나"를 확인하는 데는 작고 저렴한 비전 모델로 충분합니다. 그래서 생성과 검증을 다른 모델로 분리합니다(섹션 4의 Tool Calling에서 도구를 나눈 이유가 이것입니다). 검증 비용이 충분히 싸지면, 검사를 여러 번 해서 비싼 재생성 한 번을 막는 쪽이 이득이 됩니다. 요청 분해와 모호성 판정도 같은 논리로 경량 언어 모델에게 맡깁니다.

### ② 재시도 상한은 2회

이 설계에서 재시도 비용은 사용자 크레딧이 아니라 회사가 부담합니다. 그래서 상한 없이 돌릴 수 없습니다. 상한을 2회로 정한 근거는 확률입니다. 첫 실행에서 실패한 항목이 재시도로 성공할 확률은 회차가 갈수록 떨어집니다. 1차 재시도는 성공을 기대할 만하지만, 2차까지 실패한 항목은 모델이 그 요청 자체를 처리하지 못할 가능성이 높습니다. 3차부터는 비용만 쓰고 결과는 같을 공산이 큰 겁니다. 그 시점에는 기계를 더 돌리는 것보다 사용자에게 선택권을 돌려주는 것(화면 3)이 경험으로도 비용으로도 낫습니다. 상한 값은 고정이 아니라, 출시 후 '재시도 회차별 성공률' 데이터를 보고 재조정합니다.

### ③ 지금 할 일과 몰아서 할 일 — 실시간과 배치

카페에서 손님이 주문한 커피는 지금 만들어야 하지만, 재고 정리와 단골 노트 정리는 한가한 시간에 몰아서 하면 됩니다. 에이전트의 일도 같은 기준으로 나눕니다.

사용자가 화면 앞에서 기다리는 일 — 요청 분해, 판정, 구체화 선택지 생성, 실행, 검증 — 은 실시간으로 처리합니다. 반면 기다리는 사람이 없는 일 세 가지는 배치로 돌립니다. '결과가 달라요' 접수 건을 모아 검증 기준을 고치는 일, 재시도에 반복 실패하는 요청 유형을 분석해 지원 범위를 개선하는 일, 사용자별 선택 기록에서 취향을 승격하는 Memory 갱신(3회 규칙). 셋 다 다음 방문 전까지만 반영되면 충분해서, 호출이 저렴한 시간대에 몰아 처리합니다.

JD의 "무엇을 실시간으로, 무엇을 배치로 돌려야 할까요?"에 대한 제 답은 한 줄입니다. **사용자가 기다리는 동안의 일만 실시간으로, 다음 방문 전까지만 되면 되는 일은 전부 배치로.**

### ④ 돈이 새는지 지켜보는 계기판

마지막으로 이 구조의 비용을 감시할 숫자를 하나 정합니다. **결과 1건당 평균 호출 수.** 이 숫자가 커지면 어딘가에서 비용이 새고 있다는 뜻이고, 모호성 판정 기준을 조이거나 재시도 상한을 낮추라는 신호로 읽습니다. 사용자 크레딧을 아끼는 설계가 회사 원가를 태우는 설계가 되지 않도록, 사용자 지표(섹션 2)와 원가 지표를 같은 대시보드에서 봅니다.

## 부록 — 그 외 관찰한 UX 개선점

이번 과제와 별개로, 사용 중 발견한 개선점 하나를 남깁니다. 생성 비율(4:5, 1:1 등) 선택 시 숫자만으로는 결과물의 형태를 체감하기 어렵습니다. 비율을 탭하면 미리보기 프레임이 해당 비율로 바뀌어 보이는 방식이면, 생성 전에 결과의 쓰임새(피드용, 썸네일용)를 판단하기 쉬워질 것입니다.
